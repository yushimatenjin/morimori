# PMリリース運用統一（Morimorimori）

基準日: 2026年3月15日

## 目的

- 毎回のリリースで「詰め込み」ではなく、同じ判断軸で品質を上げる
- 主導線の成功率を最優先にし、任意機能で導線を壊さない
- 開発・デザイン・PM・CEOの意思決定を1つのループに統一する
- 今回の改善と次フェーズの期待を同時に成立させる

## 北極星（North Star）

主導線は固定:

1. 地点を決める（ギャラリー選択 or 手動検索）
2. 地形を生成する
3. GLBを保存する

この3ステップの完了率を最上位KPIとする。

## フェーズ定義（未来接続）

- Phase-1（2026公開）: 個人が最短で生成して持ち出せる
- Phase-2（2026拡張）: 生成結果を比較し、チーム合意へ繋げる
- Phase-3（2027以降）: 制作ワークフロー全体を接続する

各リリースは、どのフェーズ価値に寄与するかを必ず明記する。

## リリース前チェック（Ship Gate）

- Gate-1 導線: 主導線3ステップが最短で完了できる
- Gate-2 状態: `splash/idle/searching/ready-to-generate/generating/generated/error` が視覚的に区別できる
- Gate-3 主CTA: 画面ごとに主CTAは1つ
- Gate-4 文言: 現在地・次行動が文言で伝わる
- Gate-5 回帰: 結合テスト表の主要ケース（IT-01/02/05/06/07）がPass
- Gate-6 機能: `?review=1` で生成成功率・保存成功率・エラー復帰回数を確認できる
- Gate-6.5 必要度: 総合改善必要度が 0〜69% の範囲で推移している
- Gate-7 未来接続: `generated` 状態で次章予告があり、主CTAを阻害しない

1つでもFailなら `no-ship`。

## フィードバックループ（毎リリース）

1. 計測確認
2. ボトルネック特定
3. 改善仮説を1つ決定
4. UI実装
5. Playwright実動作確認
6. Ship/No-ship判定
7. 学習を次回へ反映

## フィードバックループ（公開後）

- 24時間: 致命的導線破綻とエラー復帰率を確認
- 72時間: 主導線完了率と保存到達率の初動を確認
- 7日: テーマ継続/縮小/ロールバックを決裁

各チェックポイントで `継続 / 縮小 / ロールバック` の判定を残す。

## 1リリース1改善ルール

- 1回のリリースで主導線に効く改善は最大1テーマ
- テーマが複数ある場合は、完了率への影響が大きい順で分割
- 「見た目改善のみ」は単独テーマにしない（導線改善とセット）

## 入力負荷の制限（UI予算）

- 初期表示で見せる入力は最大3群まで
- 追加入力は `詳細設定` として折りたたむ
- 生成後の任意機能は `仕上げ（任意）` に隔離

## 決裁権限と責任

| 項目 | Responsible | Accountable | Consulted | Informed |
|---|---|---|---|---|
| Ship/No-ship | PM | PM | CEO, Design, Dev | 全員 |
| 色味・表現トーン | Design | CEO | Storytelling, PM | Dev |
| 主導線/任意導線境界 | Storytelling | PM | Design, Dev | CEO |
| UI配置と情報階層 | Design | Design | PM, Storytelling | Dev |
| 状態遷移実装と回帰 | Dev | Dev | PM, Design | CEO |

## 役割別責務

- CEO: 色味・印象品質とブランド整合の最終承認
- PM: Ship/No-ship、主CTA優先度、リリーステーマ選定
- ストーリーテラー: 章立て、未来予告文、主導線保護
- デザイン: 情報階層、状態ごとの視認性、入力負荷の最適化
- 開発: 状態遷移の一貫性、DOMフック整合、回帰テスト維持

## 次回からの必須成果物

- 結合テスト表更新（Pass/Failとメモ）
- 変更した主導線ステップの前後比較
- 次回改善テーマ（1つ）の明文化
- `generated` 状態での未来予告文言と離脱率確認メモ

## 当日P0実行メモ（2026-03-15）

- 課題: `idle` の主CTAが状態遷移を生まず、主導線開始率を押し下げる
- 対応: 主CTA押下でおすすめ地点を自動選択し `ready-to-generate` へ遷移
- 期待効果: `idle -> ready-to-generate` の到達率改善（主CTA経由）
- 計測項目: 主CTA押下回数、`ready-to-generate` 到達回数、生成開始回数
