# UI 基盤設計

この Skill は、UI の方向性をゼロから決めるとき、見た目を立て直すとき、命名と情報設計を揃えるときに使います。

最初に `../../AGENTS.md` を読みます。コンポーネント化まで視野に入れる場合は
`../frontend-component-implementation/references/component-model.md` も読みます。

## 基盤設計の手順

1. まず画面の目的を一文で定義します。
2. ユーザーが通る状態遷移を洗い出します。
3. 各状態で必要な情報を並べます。
4. 情報を「主行動」「補助行動」「状態表示」「補足情報」に分けます。
5. 繰り返し現れる要素だけを、共通コンポーネント候補として抽出します。
6. そのあとで見た目を整えます。
7. デスクトップとモバイルは別々に確認します。

## デザイン原則

- プロダクト文言は日本語を基本とします。
- 文字組み、余白、CTA の強弱だけで階層が伝わる状態を目指します。
- 装飾は、弱いレイアウトを隠すためではなく、状態の意味を補強するために使います。
- 生成前に価値を出さないパネルは、意味のある empty state へ置き換えます。
- 状態やフィードバックが見えない UI を、デザイナーは許可しません。

## コンポーネントの考え方

コンポーネントは、先に名前を決めてから当てはめるのではなく、実際に繰り返す役割から定義します。

- 役割が同じなら同じ名前にする
- 見た目が違っても役割が同じなら同じ系統に寄せる
- 見た目が似ていても役割が違うなら別物として扱う

最低でも次の観点で命名を決めます。

- 行動なのか
- 入力なのか
- 状態表示なのか
- 要約情報なのか
- 補助説明なのか

## 出力の持ち方

レビューや提案は、次の粒度で話します。

- どの状態に効くか
- どの共通コンポーネント候補に効くか
- 文言の問題か
- 階層の問題か
- デスクトップとモバイルへどう効くか
