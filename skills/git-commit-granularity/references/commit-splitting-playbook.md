# Commit Splitting Playbook

## 1. 変更の分解テンプレート

各ファイルに対して次を埋める。

- 目的: 何を直したか
- 種別: `behavior` / `refactor` / `copy` / `build`
- 依存: どの変更に依存するか
- 単体 revert 可否: 単独で戻して壊れないか

この 4 項目が一致するものを同一コミット候補にする。

## 2. コミット境界の判断基準

同じコミットにまとめてよい条件:

- 同一のユーザー価値に寄与する。
- 同時にレビューしないと正否判定できない。
- 同時に戻す必要がある。

別コミットに分ける条件:

- どちらかが準備作業（リネーム、移動、整形）である。
- 片方だけ先に取り込んでも意味がある。
- テスト観点が異なる。

## 3. 実行コマンド例

```bash
git status --short
git diff --stat

# コミット候補Aをステージ
git add src/state.js src/ui/status-panel.js
git diff --staged

# 混ざったら戻す
git restore --staged src/ui/status-panel.js

# 候補Aをコミット
git commit -m "fix(state): generating完了時の遷移漏れを修正"
```

## 4. よくある誤り

- 「ついでにやった修正」を同コミットに含める。
- レビューしにくい巨大コミットを「後で分割する前提」で作る。
- 差分確認なしでコミットして目的外変更を混ぜる。

## 5. 最終チェック

- コミットごとに説明文を 1 文で言える。
- 各コミットに目的外ファイルが入っていない。
- 履歴順に読むと依存関係が前から後ろへ流れる。
- `git log --oneline` で連続した物語になっている。
