# Git Commit Granularity

この Skill は、変更内容をレビューしやすい単位に分割してコミットするための実務手順を提供します。

最初に `git status --short` と `git diff --stat` で全体像を把握し、必要なら `references/commit-splitting-playbook.md` を開いて分割パターンを参照します。

## 実行手順

1. 変更ファイルを一覧化し、各ファイルに「何を変えたか」を 1 行で付与します。
2. 次の軸で変更を分類します。
- 挙動変更（機能追加・バグ修正・状態遷移変更）
- 非挙動変更（リファクタ・命名整理・コメント修正）
- 表示文言変更
- ビルド生成物・依存更新
3. 1 つのコミットに 1 つの意図だけを残すように分割案を作ります。
4. 分割案ごとに `git add <path...>` でステージし、`git diff --staged` で意図の純度を確認します。
5. 混在があれば `git restore --staged <path...>` で戻して再ステージします。
6. コミット順序を「土台 -> 挙動変更 -> 補助調整」に並べて `git commit` します。
7. 最後に `git log --oneline -n <コミット数>` で読み順を確認します。

## 粒度の判定ルール

- 「レビューで 1 つの質問に答えられるか」を単位基準にします。
- ファイル数ではなく意図の数で分けます。
- 同一ファイルでも意図が違う変更は別コミットに分離します。
- 挙動変更と整形のみ変更は必ず分離します。
- 失敗時に `git revert` しやすい単位を優先します。

## 禁止パターン

- 機能追加と大規模整形を同一コミットへ混在させる。
- UI 文言修正とロジック変更を同一コミットへ混在させる。
- 生成物だけの差分を、元ソース変更コミットへ無条件で混ぜる。
- コミットメッセージで変更意図を説明できない単位でまとめる。

## コミットメッセージ規約

メッセージは次の形式を基本にします。

```text
<type>(<scope>): <要約>
```

`type` は `feat` / `fix` / `refactor` / `docs` / `chore` を使います。

例:

- `fix(state): generating から generated への遷移漏れを修正`
- `refactor(ui): status panel の責務を分割`
- `docs(skill): commit 分割プレイブックを追加`

## このプロジェクトでの使い分け

- 状態遷移に関わる変更は状態単位で分割します（例: `searching` 修正と `generating` 修正を分離）。
- UI 基盤化（`c-*`, `l-*`, `is-*` への寄せ）と挙動変更は分離します。
- `index.html` の構造変更、`styles.css` の再利用パターン変更、`app.js` の状態配線変更は同時に触っても「同一目的」かを厳格に判断します。

## 完了条件

- 各コミットが 1 つの目的に収束している。
- `git diff --staged` を見たとき、目的外の差分がない。
- コミット列を上から読むだけで変更ストーリーが追える。
- 直前コミットだけを `revert` しても破綻しにくい。
