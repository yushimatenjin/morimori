# フロントエンド コンポーネント実装

この Skill は、このプロジェクトの UI を実装またはリファクタするときに使います。

最初に `../../AGENTS.md` を読み、そのあと `references/component-model.md` を読みます。

## 実装手順

1. どのプロダクト状態を変えるのかを特定します。
2. どの UI 要素が繰り返し利用されるのかを特定します。
3. まず共通コンポーネント候補を正規化します。
4. 次に JS 側で状態遷移を明示的に配線します。
5. そのあとで画面レベルのマークアップを組みます。
6. 最後にデスクトップとモバイルを確認します。

## 実装ルール

- 画面の一部だけを場当たり的な class 追加でつないでいかないこと。
- JS から、隠されている要素、削除済みの要素、意味を失った要素を参照しないこと。
- プロダクト文言の言語を雑に混在させないこと。
- リファクタ後に古いコメントや死んだセレクタを残さないこと。
- UI 状態があるなら、見た目の偶然ではなく明示的な状態として表現すること。

## コンポーネント指向ルール

- 画面専用スタイルより、まず再利用できる部品化を優先します。
- コピペしたボタンクラスより、バリアント化を優先します。
- 見た目 class の query より、共有データフックを優先します。
- 巨大なイベント寄せ集めより、小さな責務分割モジュールを優先します。

## 最低限の分割方針

より大きなフロントエンド基盤へ移行するまでは、最低でも次の責務に寄せていきます。

- `index.html`: 構造と意味づけのコンテナ
- `styles.css`: トークンと再利用スタイル
- `app.js`: state machine / data providers / viewer / UI bindings

UI がさらに増えるなら、`app.js` は次の単位まで分けます。

- `state`
- `ui`
- `viewer`
- `services`
- `components`

## 修正時の優先順位

修正は次の順で行います。

1. 状態設計を直す
2. 状態ごとの表示内容を直す
3. 共通化すべき部品を切り出す
4. 最後に見た目の磨きを入れる

## 完了条件

次を満たさない限り、フロント実装は完了扱いにしません。

- コンポーネント命名が component model と一致している
- 対象状態が明示されている
- 主ボタンの優先順位が崩れていない
- 死んだ UI 配線がない
- デスクトップとモバイルの両方で意味が通る
