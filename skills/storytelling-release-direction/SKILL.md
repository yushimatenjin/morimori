# ストーリーテリング リリース設計

この Skill は、仕様を書く前に「哲学」と「章立て」を固定するために使います。

最初に `../../AGENTS.md` を読みます。
そのあと `../../docs/storytelling-release-playbook.md` を参照します。

## 目的

- 体験を「状態名」ではなく「章」で設計する
- 章ごとに 1 主 CTA の原則を守る
- 最優先導線を `地形生成 → 3Dモデルダウンロード` に固定する

## 設計順序（必須）

1. 哲学を 3-5 行で定義する
2. 体験を章立てに分解する（探す/作る/持ち出す/仕上げる）
3. 折りたたみ方針を決める（何を今見せて何を隠すか）
4. 章を内部状態へマッピングする
5. 各章の主 CTA と状態文言を定義する
6. 指標を設定して `ship/no-ship` 判定する

## 折りたたみルール

- 今の章に不要な設定は折りたたむ
- 次章の設定を先出ししない
- 主 CTA 以外は弱い階層に置く
- 任意機能は `generated` 後に分離する

## 状態の扱い

- `splash` `idle` `searching` `ready-to-generate` `generating` `generated` `error` は内部実装として維持する
- ユーザー向け表示は章タイトルを優先する
- 状態名をそのまま UI 文言として出さない

## 出力形式

1. 哲学
2. 章立て
3. 折りたたみ方針
4. 内部状態マッピング
5. 章別 CTA/文言
6. 機能要件（必須/任意）
7. 指標
8. リリース判定
