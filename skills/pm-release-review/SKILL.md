# PM リリースレビュー

この Skill は、UI 変更を PM 観点でレビューし、止めるか通すかを判断するときに使います。

最初に `../../AGENTS.md` を読みます。コンポーネント設計まで論点に入る場合は
`../frontend-component-implementation/references/component-model.md` も読みます。

## レビュー手順

1. まず画面を次の状態で確認します:
   - splash
   - splash を閉じた直後の idle
   - searching または configuring
   - generating
   - generated
   - mobile の idle / generated
2. 詳細を書く前に `ship` か `no-ship` を決めます。
3. 指摘は必ず重い順に出します。
4. 指摘は見た目の好みではなく、ユーザー行動と状態遷移に結びつけます。
5. コードがある場合は、根本原因になっている可能性が高いファイルと行を示します。

## PM 基準

- ユーザーは常に「次に何をするか」がわかること。
- 画面には、ひとつだけ明確な主ボタンがあること。
- 状態遷移が偶然ではなく、意図して設計されたものに見えること。
- 空白は意味があるか、明示的な empty state で説明されていること。
- 文言は対象ユーザーに合っていること。基本は日本向けサービス文言です。
- PM の目に触れる UI が、前の方針の残骸や作りかけに見えないこと。

## 必ず止める論点

- 理由のない日本語と英語の混在
- JS が書き込んでいるのに見えない状態表示
- 途中で消した痕跡が残るコメントや中途半端な UI
- 何が起きたか説明しない success 状態
- 進行状況や次アクションを示さない loading 状態
- 遷移後に意味のない白場が広がるデスクトップ画面
- モバイルで文脈や CTA 優先順位が崩れている画面

## 出力形式

返す内容:

1. verdict: `ship` または `no-ship`
2. 指摘一覧を重い順で
3. リリース前に最低限直すべきこと
