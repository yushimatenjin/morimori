# Morimorimori プロジェクトガイド

## 目指す品質

このプロジェクトは、国土地理院の標高データを使った日本向け地形生成サービスです。
目標は「動くデモ」ではなく、「そのまま公開できるプロダクト」です。すべての変更は、
わかりやすさ、状態遷移、信頼感、保守性を上げる方向で行ってください。

## 役割分担

- PM はリリース可否、状態遷移、CTA の優先順位、文言品質を持ちます。
- デザイナーは情報設計、見た目の統一感、UI 基盤の設計を持ちます。
- フロントエンドエンジニアはコンポーネント化、状態配線、場当たり的な UI ロジックの除去を持ちます。

役割別のプレイブック:

- `skills/pm-release-review/SKILL.md`
- `skills/ui-foundation-design/SKILL.md`
- `skills/frontend-component-implementation/SKILL.md`
- `skills/storytelling-release-direction/SKILL.md`

## 絶対ルール

- 明確にバイリンガルな画面でない限り、同一画面で日本語 UI と英語 UI を混在させないこと。
- 何も起きていないように見える空白状態を残さないこと。主要状態では常に「何が見えていて、次に何をすべきか」が伝わること。
- JavaScript が書き込む状態表示を、見えない場所へ追いやらないこと。
- `Removed ...` のような途中経過コメントを本番コードへ残さないこと。消すなら完全に消し、戻すなら責任を持って戻すこと。
- 同じ役割の UI がすでにあるのに、画面専用の一発クラスと一発ロジックを増やさないこと。
- 新しい状態を増やすなら、状態名、表示内容、主要 CTA、状態文言、抜け方まで定義すること。

## 必須の状態設計

UI の変更では最低でも次の状態を扱ってください。

1. `splash`
2. `idle`
3. `searching`
4. `ready-to-generate`
5. `generating`
6. `generated`
7. `error`

各状態について、必ず次を定義します。

- 何が表示されるか
- 主ボタンは何か
- 状態文言で何を伝えるか
- デスクトップとモバイルでどう見えるか

## UI 基盤の考え方

このプロジェクトには、まだ固定された UI キットはありません。部品名を先に決め打ちするのではなく、
プロダクトの状態と導線から逆算して UI 基盤を設計します。

設計順序:

1. 画面の目的を定義する
2. 状態遷移を定義する
3. 情報の優先順位を決める
4. 繰り返し出る操作と表示を抽出する
5. そのあとで共通部品名とバリアントを決める

最初から大量の部品名を固定しないでください。同じ役割の UI が 2 回以上出ると確認できた段階で、
共通部品候補として扱います。

## 実装方針

- `index.html` は構造と意味づけを担う場所に保つこと。
- `styles.css` は画面専用 CSS より先に、トークンと再利用パターンを整えること。
- `app.js` は state / data / viewer / UI binding の責務に寄せていくこと。
- JS から参照する UI 要素は、その時点の表示状態で存在し、意味を持っていること。
- 主要な UI 変更では、必ずデスクトップとモバイルを両方確認すること。
- 見た目の差分だけで済ませず、状態遷移と文言も同時に見直すこと。

## 修正方針

修正は次の順で進めます。

1. 何が壊れているかを状態単位で特定する
2. その状態で必要な情報と CTA を定義し直す
3. 再利用すべき UI を抽出する
4. 構造、見た目、挙動の順で直す
5. デスクトップとモバイルで確認する

「なんとなく見た目を整える」修正は禁止です。必ず次を説明できる形で直します。

- どの状態を直したか
- 何が改善されたか
- どの共通部品に寄せたか
- どの回帰を確認したか

## 命名規約

ユーザー向け文言は日本語を基本とします。内部識別子とコンポーネント名は英語で統一します。
ただし、名前は見た目ではなく役割で決めます。

標準ルール:

- JS のコンポーネント名・クラス名: `PascalCase`
- JS の変数名・関数名: `camelCase`
- CSS のコンポーネントブロック: `c-*`
- CSS のレイアウトオブジェクト: `l-*`
- CSS の状態フラグ: `is-*`, `has-*`
- DOM の挙動フック: `data-component`, `data-action`, `data-state`

例:

- `c-button c-button--primary`
- `c-panel`
- `c-summary-card`
- `is-loading`
- `has-data`
- `data-action="generate-terrain"`

見た目のための class を、そのまま挙動フックとして使わないでください。

## リリース判定基準

次の条件を満たさない限り、リリース可能とはみなしません。

- splash、idle、loading、success、error が見た目でも区別できる
- 各主要状態で、主ボタンが一目でわかる
- 死んだ UI、見えない状態表示、説明不能な空白がない
- 右レール、ヒント、要約情報は「出す意味があるときだけ」出る
- コンポーネント名とバリアントが、その時点の UI 基盤定義と一致している
