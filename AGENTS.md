# Morimorimori プロジェクトガイド

## 目指す品質

このプロジェクトは、国土地理院の標高データを使った日本向け地形生成サービスです。
目標は「動くデモ」ではなく、「そのまま公開できるプロダクト」です。すべての変更は、
わかりやすさ、状態遷移、信頼感、保守性を上げる方向で行ってください。

## 2026 リニューアル宣言（基準日: 2026年3月15日）

2026年以降の Morimorimori は、地形プレビューサイトではなく、
「制作現場でそのまま使える背景地形制作ワークスペース」として設計します。

固定する体験の主語:

- 探す: どこを作るかを決める
- 作る: 条件を決めて地形を生成する
- 持ち出す: 3D モデルを保存して制作へ接続する
- 仕上げる: 必要な場合のみ視点調整を行う

主導線は `探す → 作る → 持ち出す` で固定し、`仕上げる` は任意導線として分離します。

未来フェーズの扱い:

- Phase-1（2026公開）: 個人制作を最短で完了
- Phase-2（2026拡張）: 比較・共有でチーム判断を高速化
- Phase-3（2027以降）: 制作ワークフロー全体へ接続

`generated` 状態では「現在の価値 → 次章の価値」の順で予告を出し、
主 CTA（3Dモデル保存）を最優先に保ちます。

## 絶対ルール

- 明確にバイリンガルな画面でない限り、同一画面で日本語 UI と英語 UI を混在させないこと。
- 何も起きていないように見える空白状態を残さないこと。主要状態では常に「何が見えていて、次に何をすべきか」が伝わること。
- JavaScript が書き込む状態表示を、見えない場所へ追いやらないこと。
- `Removed ...` のような途中経過コメントを本番コードへ残さないこと。消すなら完全に消し、戻すなら責任を持って戻すこと。
- 同じ役割の UI がすでにあるのに、画面専用の一発クラスと一発ロジックを増やさないこと。
- 新しい状態を増やすなら、状態名、表示内容、主要 CTA、状態文言、抜け方まで定義すること。
- 主 CTA より目立つ副操作を置かないこと。任意機能は主導線完了後に見せること。

## 必須の状態設計

UI の変更では最低でも次の状態を扱ってください。

1. `splash`
2. `idle`
3. `searching`
4. `ready-to-generate`
5. `generating`
6. `generated`
7. `error`

各状態について、必ず次を定義します。

- 何が表示されるか
- 主ボタンは何か
- 状態文言で何を伝えるか
- デスクトップとモバイルでどう見えるか

## 2026 色味・表現指針（CEO 監修）

色は装飾ではなく、状態理解と行動優先度を伝えるために使います。

基準パレット:

- `Night Base`: `#0b1120`（背景の基調）
- `Cloud Text`: `#e2e8f0`（本文）
- `Terrain Cyan`: `#22d3ee`（主導線の強調）
- `Sun Amber`: `#f59e0b`（処理中・注意）
- `Forest Emerald`: `#10b981`（完了）
- `Alert Vermilion`: `#f43f5e`（エラー）

状態と色の対応:

- `splash`: Night Base + Terrain Cyan（開始の期待）
- `idle` / `ready-to-generate`: Terrain Cyan（次行動の明確化）
- `searching` / `generating`: Sun Amber（進行中の可視化）
- `generated`: Forest Emerald（成功と保存誘導）
- `error`: Alert Vermilion（原因提示と再開導線）

禁止事項:

- 状態と無関係なアクセント色追加
- 主 CTA と副 CTA の同強度表現
- 「雰囲気だけ」のグラデーション追加

## 2026 オプション設計と UI 配置

オプションは「増やす」ではなく「順番で出す」を原則にします。

- 初期表示は主導線に必要な入力のみ（地点・範囲・生成）。
- `2地点モード` を追加する場合、開始地点と終点の2入力だけで完結させ、範囲は自動計算にする。
- 任意機能は `generated` 到達後に開示し、折りたたみを初期値にする。
- 同時に見せる入力群は最大3群までに抑える。
- デスクトップは左に制御、右にビューアを固定し、主 CTA は常に視界内へ置く。
- モバイルは下部固定 CTA を優先し、補助情報は CTA より下に追い出さない。

状態ごとの配置契約:

| 状態 | 何を表示するか | 主ボタン | 状態文言 | デスクトップ | モバイル |
|---|---|---|---|---|---|
| `splash` | 体験概要3行と開始導線 | 制作を始める | 何をすると何が得られるか | 中央オーバーレイ | 全画面オーバーレイ |
| `idle` | 地点ギャラリーと検索導線 | 地点を選ぶ/検索する | まず地点選択が必要 | 左パネル先頭に配置 | 地点選択を最上段 |
| `searching` | 検索進捗と待機理由 | 検索中（非活性） | 検索中であること | 状態ピルと詳細を固定 | テキスト短文化 |
| `ready-to-generate` | 生成設定サマリー | この条件で地形を生成 | 条件確認後に生成 | 生成セクションを中央寄せ | 主 CTA を固定表示 |
| `generating` | ローディングオーバーレイ | 生成中（非活性） | 進捗と想定待ち時間 | ビューア上に重ねる | 全面ロック表示 |
| `generated` | 保存導線と成果サマリー | 3Dモデルを保存 | 先に保存、その後任意調整 | 保存 CTA を最上位 | 下部固定 CTA を保存に切替 |
| `error` | 原因/影響/次行動 | 条件を見直して再試行 | 再開方法を明示 | 状態カードで可視化 | CTA と原因を縦積み |

## 2026-03-15 UI再設計 実装契約

青寄り固定の見た目を避けるため、色は常時固定ではなく状態で切り替える。

- 状態色は `data-state` で制御し、`idle/ready` は `Terrain Cyan`、`searching/generating` は `Sun Amber`、`generated` は `Forest Emerald`、`error` は `Alert Vermilion` を使う。
- サイドバーとモバイル下部固定に同じ主 CTA を置き、状態ごとにラベル・活性/非活性を一致させる。
- 地点選択で自動生成しない。`ready-to-generate` を経由して主 CTA で生成する。
- `generated` 到達前に仕上げ系 UI を展開しない。保存導線の下に任意導線を配置する。
- エラー表示は必ず `原因 / 影響 / 次` を縦積みで表示し、主 CTA で復帰導線へ戻す。

## フィードバックループと権限・責任

決裁は「誰でも言える」ではなく「誰が決めるか」を固定します。

決裁権:

- CEO: 色味、表現トーン、ブランド整合の最終承認
- PM: `ship / no-ship` の最終決裁
- ストーリーテラー: 章立て、主導線/任意導線の境界承認
- デザイナー: 情報階層、コンポーネント見た目の承認
- フロントエンド: 実装方式、状態配線、技術的妥当性の承認

運用ループ:

1. 日次: 主要状態の破綻有無を 10 分で確認
2. 週次: 1リリース1改善テーマを決定
3. 実装後24h/72h/7day: 指標と実動作を再評価
4. 継続/縮小/ロールバックを決裁し、次週テーマへ接続

必須成果物:

- 状態別変更サマリー（何を直し、何が改善したか）
- 主 CTA 変更記録（文言・配置・理由）
- デスクトップ/モバイル確認結果
- 次サイクルへ渡す学習メモ

## UI 基盤の考え方

このプロジェクトには、まだ固定された UI キットはありません。部品名を先に決め打ちするのではなく、
プロダクトの状態と導線から逆算して UI 基盤を設計します。

設計順序:

1. 画面の目的を定義する
2. 状態遷移を定義する
3. 情報の優先順位を決める
4. 繰り返し出る操作と表示を抽出する
5. そのあとで共通部品名とバリアントを決める

最初から大量の部品名を固定しないでください。同じ役割の UI が 2 回以上出ると確認できた段階で、
共通部品候補として扱います。

## 実装方針

- `index.html` は構造と意味づけを担う場所に保つこと。
- `styles.css` は画面専用 CSS より先に、トークンと再利用パターンを整えること。
- `app.js` は state / data / viewer / UI binding の責務に寄せていくこと。
- JS から参照する UI 要素は、その時点の表示状態で存在し、意味を持っていること。
- 主要な UI 変更では、必ずデスクトップとモバイルを両方確認すること。
- 見た目の差分だけで済ませず、状態遷移と文言も同時に見直すこと。
- UI や状態を変更したら、同一 PR で `AGENTS.md` と関連ドキュメントを更新すること。

## 修正方針

修正は次の順で進めます。

1. 何が壊れているかを状態単位で特定する
2. その状態で必要な情報と CTA を定義し直す
3. 再利用すべき UI を抽出する
4. 構造、見た目、挙動の順で直す
5. デスクトップとモバイルで確認する

「なんとなく見た目を整える」修正は禁止です。必ず次を説明できる形で直します。

- どの状態を直したか
- 何が改善されたか
- どの共通部品に寄せたか
- どの回帰を確認したか

## 命名規約

ユーザー向け文言は日本語を基本とします。内部識別子とコンポーネント名は英語で統一します。
ただし、名前は見た目ではなく役割で決めます。

標準ルール:

- JS のコンポーネント名・クラス名: `PascalCase`
- JS の変数名・関数名: `camelCase`
- CSS のコンポーネントブロック: `c-*`
- CSS のレイアウトオブジェクト: `l-*`
- CSS の状態フラグ: `is-*`, `has-*`
- DOM の挙動フック: `data-component`, `data-action`, `data-state`

例:

- `c-button c-button--primary`
- `c-panel`
- `c-summary-card`
- `is-loading`
- `has-data`
- `data-action="generate-terrain"`

見た目のための class を、そのまま挙動フックとして使わないでください。

## リリース判定基準

次の条件を満たさない限り、リリース可能とはみなしません。

- `splash / idle / searching / ready-to-generate / generating / generated / error` が見た目でも区別できる
- 各主要状態で、主ボタンが一目でわかる
- 死んだ UI、見えない状態表示、説明不能な空白がない
- 右レール、ヒント、要約情報は「出す意味があるときだけ」出る
- コンポーネント名とバリアントが、その時点の UI 基盤定義と一致している
- 24h/72h/7day のフィードバックループ記録が残っている
